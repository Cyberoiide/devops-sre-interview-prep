# q224 — Ingress 5xx error codes (500 / 502 / 503 / 504)

## The interview question

> "You're on call. Users report errors and you see 5xx responses coming back
> from your NGINX Ingress. Walk me through what **500**, **502**, **503**, and
> **504** each tell you, how you tell them apart, and how you'd confirm the
> cause from the cluster."

This is a classic SRE triage question. The interviewer wants to see that you
understand the difference between an error the **application generated** and an
error the **edge/proxy generated on the application's behalf** — and that you
know exactly which command confirms each one.

## What you should be able to answer

- What each of 500, 502, 503, 504 means when it comes back through an Ingress.
- The one-line mental model that separates them (app error vs. no backend vs.
  reached-but-timed-out vs. reached-but-bad-response).
- Which of these are generated by the **ingress controller itself** vs. simply
  **proxied through** from the backend.
- How to confirm each cause: `kubectl get endpoints`, controller access/error
  logs, upstream status vs. returned status, timeout annotations.
- The default NGINX Ingress proxy timeout and how to change it.
- Why a rolling deploy commonly produces a burst of 502s or 503s.
- How readiness probes relate directly to 503.

## The 15-second mental model

| Code | Meaning at the edge | One-liner |
|------|--------------------|-----------|
| **500** | The app itself returned an error | "The app answered, and its answer was 'I broke.'" |
| **502** | Reached a backend, got an unusable response | "I reached it, but the answer was garbage / it hung up." |
| **503** | No healthy backend to send to | "There is nowhere to send this." |
| **504** | Reached a backend, it never answered in time | "I reached it, it never answered in time." |
